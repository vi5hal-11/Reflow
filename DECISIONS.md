## 2026-07-20 â€” Phase 4 (Google Calendar, bidirectional)

- **Token storage is service-role-only**: `calendar_connections` (migration 0004, applied) has RLS enabled with deliberately **zero policies** â€” the browser can never read refresh tokens; the only path is `lib/supabase/admin.ts` (secret key, guarded against client-bundle import). - **Own OAuth flow, not Supabase's Google provider**: a dedicated `/api/calendar/connect â†’ Google â†’ /api/calendar/callback` round-trip with an httpOnly SameSite=Lax state cookie (10-min TTL, single-use), `access_type=offline&prompt=consent` so a refresh token is guaranteed (a connection without one is refused). Callback outcomes are always redirects to `/today` â€” failures land on `?calendar_error=1` and show a calm "try again whenever" notice.
- **The anti-deadlock rule** (the load-bearing correctness decision): blocks Reflow pushes to Google carry a `reflow_task_id` private extended property, and the sync pull **excludes** any event carrying it from the `calendar_events` cache. Without this, every pushed block would return as a fixed block and the scheduler would re-flow around its own output.
- **Push is post-response** via Next 16's `after()`: plan latency never pays for Google round-trips; push failures cost only the mirror (insert on first placement, patch on move, delete on overflow/unschedule; a 404/410 on patch means recreate). Kept/unmoved blocks are never re-pushed.
- **Pull is strictly window-scoped** (client sends local yesterday â†’ +7d): cache rows outside the window are never touched; all-day events are dropped (no dateTime), `transparency=transparent` events are cached with `is_busy=false` (the plan route only treats `is_busy=true` as fixed); `changed` is exact (upserts only for rows that genuinely differ) so the client re-flows only when the outside world actually moved.
- **UI restraint**: calendar UI renders only when the feature is configured (`available`); sync runs on mount + tab focus throttled to 5 minutes (manual "sync" bypasses and resets the throttle); sync 409 quietly reverts to the connect affordance; disconnect deletes pushed Google events best-effort, then always succeeds locally.

## 2026-07-19 â€” Phase 3 (deterministic auto-scheduler + self-heal)

- **Contract refinement**: the engine takes `energy_windows` as concrete datetime ranges (resolved by the web client against its local day) instead of Â§5's clock-string `energy_profile` â€” the engine stays timezone-agnostic and purely interval-based. `FlexibleTask` carries `scheduled_start/end` (for stable re-flow) and `created_at` (FIFO tiebreak); `PlacedBlock.kept` tells the client which blocks didn't move.
- **Energy semantics**: a task *starts* inside a matching energy window even if it spills past the window's end (start deep work in deep hours), preferring the earliest energy-matched start; falls back to earliest fit. Deliberate: anchoring the start matters more than containment.
- **Stability semantics**: an existing placement is kept verbatim iff it lies within the remaining window (or is *in progress* â€” started, not yet ended, at `now`) and conflicts with nothing. A block missed entirely (ended before `now`) is rescheduled â€” that's the self-heal. Kept blocks win by earlier `scheduled_start` (then id) when they conflict with each other.
- **Wildcard strategy**: reserved at the *end of the day* (latest fitting free space), after Big 3 placement but before everything else â€” Big 3 outrank wildcards, wildcards outrank the rest. Wildcards are ephemeral (returned per plan, rendered, not persisted â€” no schema for them until a real need appears) and rendered click-through so manual placement can override breathing room.
- **Self-heal triggers (web)**: re-flow fires after complete/undo and manual place, and on tab focus â€” but only once the user has planned at least once this session, throttled to one auto re-flow per 60s, and always silent on failure. **Unplace deliberately does NOT trigger a re-flow**: the engine would immediately re-place the task (deterministically, likely in the same spot), fighting the user's explicit choice.
- **Ranking**: `(not is_big3, deadline is None, deadline, priority, created_at is None, created_at, id)` â€” id as the final tiebreak makes every sort fully deterministic.
- **Measured perf**: 0.47ms warm re-flow for 50 tasks + 8 fixed blocks (target <50ms, Â§5). 41 tests green: unit + nasty cases, hypothesis property tests (invariants Ã—200 examples, determinism, stability round-trip), perf guard at 200ms to stay CI-unflaky.
- **Parallel-agent build note**: Phase 3 was attempted as a two-stream agent workflow (engine âˆ¥ web); the agents hit the session usage limit mid-build. The web agent's finished `/api/plan` route + types survived and were kept; the engine, tests, and `/today` integration were completed inline. The adversarial-verify pass planned for the workflow was folded into the property-test suite instead.
